Trinity Launcher

Modular graphical environment for Minecraft Bedrock on Linux

Visit our official website
for step-by-step installation guides.

BSD 3-Clause Qt 6 GitHub Flatpak AppImage Docker

Trinity Launcher is a modern, modular graphical environment to run and manage Minecraft: Bedrock Edition on Linux, macOS and more. Designed to run natively as well as inside Flatpak, AppImage and Docker, it uses Qt6 and follows a clean architecture based on separated libraries (TrinityCore and TrinityUI).

Key features:

  • Multi-version management: extract and organize different game versions (APKs).
  • Content Manager (Trinito): centralized interface for mods, textures, shaders and worlds.
  • Native integration: Flatpak, AppImage, Docker and local execution.
  • Discord Rich Presence: natively implemented.

System Dependencies

Trinity Launcher requires a wide range of system dependencies to function correctly, including:

  • Graphics: Mesa, Vulkan, EGL, GL, DRM, GBM, Wayland, X11
  • Audio: ALSA, PulseAudio, PipeWire, JACK, sndio
  • Input: libevdev, libusb, libinput, X11 extensions
  • Network: libcurl, OpenSSL, D-Bus, BlueZ
  • Compression: libzip, libpng
  • Internationalization: IBus, libxkbcommon

Project Architecture

Library Structure

The project is divided into two main libraries:

  • TrinityCore Business logic (version management, configuration, Bedrock runtime communication, pack management, launching and export).
  • TrinityUI Qt-based graphical interfaces (windows, dialogs, widgets).

Clear Separation of Responsibilities

  • src/TrinityLib/core/ File and configuration logic
  • src/TrinityLib/ui/ User interface components
  • app/ Main binary trinity
  • resources/ Resources: branding, icons, shortcuts, i18n, fonts, runtime
Modular architecture: Facilitates maintenance, testing and component reuse.

Trinchete

Launcher Interface

Top Bar

  • + Extract APK: select an .apk, name it and extract with mcpelauncher-extract.
  • Import: restore a saved version from .tar.gz.
  • Tools: opens the trinito.

Right Panel (selected version)

  • PLAY: runs mcpelauncher-client -dg <path> and closes the launcher.
  • Create Shortcut: generates a .desktop file in ~/Descargas/.
  • Edit Settings: add environment variables (e.g.: DRI_PRIME=1).
  • Export: saves the version + its data as .tar.gz.
  • Delete: permanently removes the version.
Modern dark-themed interface with icon support.

Building & Installation

Method 1: Automated (Recommended)

Makefile build.sh: 

# Install deps, build and run (ideal first time)
make start

# Quick rebuild after code changes
make run

# Build and install to /usr/local/bin
make install

# Clean build files
make clean

# Update translation .ts files
make translations

Method 2: Manual with CMake/Ninja 

mkdir build && cd build

# Configure with Ninja
export CC=clang
export CXX=clang++
cmake -G Ninja -DCMAKE_BUILD_TYPE=Release ..

# Build
ninja

# Run
./app/trinity

Method 3: Direct CMake 

cmake -B build -S . -DCMAKE_BUILD_TYPE=Release \
  -DCMAKE_CXX_COMPILER=clang++ -DCMAKE_C_COMPILER=clang

cmake --build build --parallel $(nproc)

./app/trinity
Prerequisite: The mcpelauncher-client and mcpelauncher-extract binaries must be installed. See the official repo: mcpelauncher-manifest

Isolated Environment with Docker

Using Docker Compose (Recommended) 

# Build image with user permissions
UID=$(id -u) GID=$(id -g) docker compose build

# Enter container
UID=$(id -u) GID=$(id -g) docker compose run --rm trinity

# Inside, build:
./build.sh --clean --release

Using raw Docker 

docker build --build-arg UID=$(id -u) --build-arg GID=$(id -g) \
  -t trinity-launcher .

docker run --rm -it \
  --user $(id -u):$(id -g) \
  -e DISPLAY=$DISPLAY \
  -v "$(pwd)":/project \
  -v cargo-cache:/home/trinity/.cargo/registry \
  -v rust-cache:/home/trinity/project/target \
  -v dev-v:/data \
  trinity-launcher /bin/bash

AppImage Packaging

Trinity Launcher can be packaged as an AppImage for portable distribution. The project includes GitHub Actions workflows to automate this process.

Components included in AppImage

  • mcpelauncher-client Game engine
  • mcpelauncher-extract APK extractor
  • trinity Main launcher
  • msa-daemon and msa-ui-qt Authentication services
  • Critical libraries: libzip, libevdev, libminizip

Available CI/CD Workflows

  • x86_64: simple-appimage.yml x86_64-appimage.yml
  • ARM: appimage-arm.yml arm-zink.yml
  • macOS: macosx.yml mesa-macos.yml
AppImages are generated with linuxdeploy and the Qt plugin, ensuring full portability across Linux distributions.

Flatpak Packaging

Requirements 

flatpak install flathub io.qt.qtwebengine.BaseApp//6.9
flatpak install flathub org.kde.Platform//6.9 org.kde.Sdk//6.9

Build

flatpak-builder --user --force-clean build-dir com.trench.trinity.launcher.json
flatpak-builder --repo=repo --force-clean build-dir com.trench.trinity.launcher.json
flatpak build-bundle repo trinity.flatpak com.trench.trinity.launcher
flatpak install ./trinity.flatpak
The manifest must include libevdev, libzip and copy files/ to /app.

Translations & Internationalization

.ts files:

How to add a language

  1. resources/i18n/ trinity_es.ts
  2. trinity_en.ts)
  3. Qt Linguist or a text editor
  4. resources/resources.qrc
  5. ./build.sh --release --run

Required Tools

  • Qt Linguist (Recommended)
    • Ubuntu/Debian: sudo apt install qt6-tools-dev-tools
    • Arch Linux: sudo pacman -S qt6-tools
  • Or any text editor (GEANY, nvim, VS Code, Notepad++, etc.)

Update existing translations 

make translations
# or / o usa el script:
./build.sh --update-ts
Important rules:
Do NOT translate variables like %1, %2
Keep the & symbol for keyboard shortcuts (Alt+key)
Respect the original punctuation

Contribute to the Project

Contributions are welcome! Please read these important guidelines:

Prior Considerations

Trinity Launcher is in an organizational phase. Current priority is to document, structure and report, not to optimize code.

Explicit and well-documented commits

Commits must be clear and specific: 

✅ GOOD:
git commit -m "feat: add lib validation in trinity"
git commit -m "fix: correction in data path in Flatpak"
git commit -m "docs: update build instructions"

❌ BAD:
git commit -m "changes"
git commit -m "updates"

Recommended format (Conventional Commits)

  • feat: New feature
  • fix: Bug fix
  • docs: Documentation
  • refactor: Code change without new functionality
  • test: Tests
  • chore: Maintenance tasks

Code Standards

Element Style Example
Functions/Methods camelCase loadInstalledVersions()
Variables camelCase selectedVersion
Constants UPPER_SNAKE_CASE MAX_RETRIES
Classes PascalCase LauncherWindow
Files snake_case launcher_window.cpp

Reporting Bugs

If you find a bug, do NOT fix it immediately. First:

  1. Open an Issue
  2. Include: descriptive title, expected and actual behavior, steps to reproduce, system
  3. Label as bug
  4. Wait for maintainers to decide when and how to fix it

AI Usage

AI use is allowed for support, but do NOT let AI modify code without prior documentation. Stability is more important than optimization.
